AULABS — MVP Plan
=================

PRODUCT OVERVIEW
----------------
AULABS is an automotive services platform. Drivers find workshops, request services
for their cars, and workshop owners/staff manage those requests. The end goal is a
full service history per car — a "digital passport."

MVP core loop:
  user registration (no role selection — any user can do anything based on context)
  when user registered → submit workshop → pending approval → admin approves
  when user registered → car registration → workshop discovery → service request → workshop response → service record
  when user registered → browse workshops → filter by category/city/around current location/open now → view details

WHAT ALREADY EXISTS
-------------------
- User (Devise auth, no roles yet)
- ServiceCategory (9 seeded: STO, Tire Service, Car Wash, Detailing, Evacuator, Diagnostics, Oil Change, Electric Repair, Body Repair)
- Workshop (CRUD, working hours, Active Storage for logo/photos, filtering by city/country/category/open_now)
- Public workshop index/show; auth globally enforced

KEY ARCHITECTURAL DECISIONS
----------------------------
1. There is NO operator role. Every person in the system is a User (driver by default).
   Workshop access is determined entirely by the workshop_operators join table.
   A user who submits a workshop becomes its "owner" via that join table.

2. Admin is a SEPARATE Devise model (admins table), not a User. Admins manage the
   platform (approve/decline workshops, manage users) but never appear in the users table.

3. User↔Workshop is many-to-many via workshop_operators join table.
   The join table has a role column: owner(0), staff(1).
   "Is this user a workshop manager?" = user.workshop_operators.exists?

4. Workshop has many service categories via workshop_service_categories join table
   (removing the current single service_category_id on workshops).
   Pricing and estimated service duration are declared per workshop per category on this join model.

5. Two layouts:
   - application layout: driver-facing (browse, my cars, my requests)
   - workshop layout: workshop management (incoming requests, settings)

6. Workshop self-registration: any user submits workshop details → status: pending →
   admin approves/declines → on approval the submitting user is the owner.

7. Car ownership transfers are user-to-user with a full audit trail across three tables:
   car_transfers, car_transfer_events, car_ownership_records.

8. ServiceRequest references workshop_service_category_id (not service_category_id directly).
   Price is snapshotted at request time so history is accurate even if workshop changes pricing.
   Service category is accessed via: service_request.workshop_service_category.service_category.

9. ServiceRecord is the car's "digital passport" entry. It captures the full picture of a
   completed service: summary, parts used, costs, mileage at service, next service recommendation.

10. Queue management is a first-class feature. Each workshop has a live Queue per day
    (optionally per service category). Drivers join it remotely via QueueEntry.

DATA MODEL CHANGES
------------------
REMOVE:
  - service_category_id from workshops table

ADD:
  - role column on users (integer enum: driver=0)
    Note: only driver for now; kept for future extensibility.
  - admins table (separate Devise model — platform administrators)
  - workshop_operators table (user_id, workshop_id, role: owner=0/staff=1)
  - status column on workshops (pending=0, active=1, declined=2, suspended=3)
  - decline_reason column on workshops (text, nullable — populated by admin on decline)
  - latitude, longitude columns on workshops (decimal, nullable — for proximity search)
  - workshop_service_categories table (workshop_id, service_category_id, pricing fields,
    estimated_duration_minutes)
  - cars table (includes fuel_type, odometer, engine_volume, transmission)
  - car_transfers table
  - car_transfer_events table (append-only audit log)
  - car_ownership_records table (append-only ownership timeline)
  - service_requests table (references workshop_service_category_id, price_snapshot jsonb,
    lock_version for optimistic locking)
  - service_records table (full service passport: parts_used, costs, odometer_at_service,
    next service recommendation)
  - queues table (workshop_id, service_category_id, date, status)
  - queue_entries table (queue_id, user_id, car_id, position, status, estimated_wait_minutes)

  car_transfer_id (fk, nullable)     — null = original registration; fk otherwise
  created_at

Car model:

  has_many :service_requests
  has_many :service_records, through: :service_requests
  has_many :car_transfers

  validates presence: make, model, year, license_plate, fuel_type

Note: fuel_type is required. engine_volume is optional (not applicable to EVs).
      odometer is updated by the workshop when creating a ServiceRecord.
      transmission is optional — not all service categories need it.

VIN claim flow (on create, if VIN already exists):
  1. User enters VIN → system detects existing car
  2. Block creation; show: "This vehicle is already registered. If you are the new owner — request a transfer."
  3. User initiates transfer → CarTransfer.create(status: :requested, token: SecureRandom.urlsafe_base64)
  4. car_transfer_events: transfer_requested (actor: to_user)
  5. from_user receives email with approval link (token-based)
  6. car_transfer_events: notification_sent
  7a. from_user approves →
      car.update!(user_id: to_user_id)
      car_ownership_records: previous record gets ended_at = now
      car_ownership_records: new record (to_user, started_at: now, car_transfer_id: this)
      car_transfer_events: approved (actor: from_user)
      car_transfer_events: ownership_transferred (metadata: { from_user_id:, to_user_id: })
  7b. from_user rejects →
      car_transfer.rejected!
      car_transfer_events: rejected (actor: from_user, metadata: { reason: optional })
  7c. to_user cancels →
      car_transfer.cancelled!
      car_transfer_events: cancelled (actor: to_user)
  7d. expires_at passed (cron/SolidQueue) →
      car_transfer.expired!
      car_transfer_events: expired (actor: system)

Key audit queries:
  car.car_ownership_records.order(:started_at)           — full ownership timeline
  car.car_transfers.includes(:car_transfer_events)        — all transfer attempts
  CarTransferEvent.where(event_type: :ownership_transferred) — all completed transfers in system
  CarTransfer.where(from_user: user).or(...)              — all transfers a user was involved in

CarsController (full CRUD, application layout):
  - index: current_user's cars
  - show: car details + full service history table
  - new/create/edit/update/destroy: standard with ownership guard
  - transfer flow handled by CarTransfersController

Routes:
  resources :cars do
    resources :transfers, controller: "car_transfers", only: [:new, :create]
  end
  resources :car_transfers, only: [:show] do
    member { patch :approve; patch :reject; patch :cancel }
  end

STEP 7 — Service Requests (Driver Side)
-----------------------------------------
Migration create_service_requests:
  car_id (fk), workshop_id (fk)
  workshop_service_category_id (fk)  — NOT service_category_id directly; links to the join model
                                       so we know exactly what was offered by this workshop
  price_snapshot  jsonb, nullable    — {min:, max:, unit:, currency:} captured at request time;
                                       preserved even if workshop later changes their pricing
  status (int, default 0), description (text), preferred_time (datetime)
  lock_version (int, default 0)      — optimistic locking; prevents concurrent status transitions
  Indexes: [workshop_id, status], [car_id, status]

Status enum: pending=0, accepted=1, rejected=2, in_progress=3, completed=4

ServiceRequest model:
  belongs_to :car
  belongs_to :workshop
  belongs_to :workshop_service_category
  has_one :service_category, through: :workshop_service_category
  has_one :service_record
  enum :status, { pending: 0, accepted: 1, rejected: 2, in_progress: 3, completed: 4 }
  validates presence: description, preferred_time
  validate :preferred_time_within_working_hours   — checks WorkingHour for that weekday
  validate :car_belongs_to_current_user           — scoped in controller, validated in model
  validate :service_offered_by_workshop           — confirms workshop_service_category.workshop_id
                                                    matches workshop_id; prevents impossible requests
  before_create :snapshot_price                   — copies current pricing into price_snapshot
  scope :recent, -> { order(created_at: :desc) }
  def driver = car.user

  private

  def snapshot_price
    wsc = workshop_service_category
    return unless wsc
    self.price_snapshot = {
      min: wsc.price_min,
      max: wsc.price_max,
      unit: wsc.price_unit,
      currency: wsc.currency
    }.compact
  end

  - create: validates car belongs to current_user; snapshots price automatically
  - index: all requests across current_user's cars
  - show: own requests only; shows service record if completed; shows price_snapshot

app/views/workshops/show.html.erb: add "Request Service" button

STEP 10 — Queue Management
----------------------------
A Queue is a live, date-scoped line at a workshop for walk-in or on-demand service.
Distinct from a ServiceRequest (which is a pre-booked appointment with preferred_time).
A driver joins a queue remotely and arrives when called.

Migration create_queues:
  workshop_id (fk)
  service_category_id (fk, nullable)  — nil = general queue; set = category-specific queue
  date (date, not null)               — which day this queue is for
  status (int, default 0)             — enum: open=0, paused=1, closed=2
  Unique index on [workshop_id, service_category_id, date] — one queue per workshop/category/day

Migration create_queue_entries:
  queue_id (fk)
  user_id (fk)
  car_id (fk, nullable)               — driver selects which car; optional for MVP
  position (int, not null)            — assigned on join; sequential within queue
  status (int, default 0)             — enum: waiting=0, called=1, in_service=2, completed=3, no_show=4
  estimated_wait_minutes (int, nullable) — recomputed when entries ahead are served
  joined_at (datetime, not null)
  called_at (datetime, nullable)      — when workshop called this driver
  Unique index on [queue_id, position]
  Unique partial index on [queue_id, user_id] WHERE status IN (0,1,2) — no duplicate active entries

Queue model:
  belongs_to :workshop
  belongs_to :service_category, optional: true
  has_many :queue_entries
  enum :status, { open: 0, paused: 1, closed: 2 }
  scope :today, -> { where(date: Date.current) }
  def active_entries = queue_entries.where(status: [:waiting, :called, :in_service])
  def next_position = (queue_entries.maximum(:position) || 0) + 1

QueueEntry model:
  belongs_to :queue
  belongs_to :user
  belongs_to :car, optional: true
  enum :status, { waiting: 0, called: 1, in_service: 2, completed: 3, no_show: 4 }
  after_create :recompute_wait_estimates   — recalculates estimated_wait_minutes for all waiting entries
  def recompute_wait_estimates             — uses queue.service_category.estimated_duration_minutes
                                             multiplied by position ahead

Driver flow:
  Workshop show page → "Join Queue" button (if queue open today for this category)
  POST /queues/:queue_id/entries — creates QueueEntry, returns position + estimated wait
  Driver can view position at /queue_entries/:id/show
  Workshop calls next: PATCH /workshop_management/queues/:id/entries/:entry_id/call
  On call: entry.called!, estimated_wait_minutes zeroed for this entry, recalculate others

Workshop management flow:
  WorkshopManagement::QueuesController:
    - index: today's queues for this workshop; shows open/paused/closed status
    - show: live entry list (position, status, car, wait estimate)
    - open  (PATCH member): queue.open!  — or create today's queue if not exists
    - pause (PATCH member): queue.paused!
    - close (PATCH member): queue.closed!

  WorkshopManagement::QueueEntriesController:
    - call    (PATCH): entry.called! + recalculate
    - serve   (PATCH): entry.in_service!
    - complete (PATCH): entry.completed!
    - no_show (PATCH): entry.no_show! + recalculate
    - all transitions use optimistic locking (lock_version on queue_entries)

Routes:
  resources :queues, only: [:show] do
    resources :queue_entries, only: [:create, :show]
  end
  namespace :workshop_management do
    resources :workshops, only: [] do
      resources :queues, only: [:index, :show] do
        member { patch :open; patch :pause; patch :close }
        resources :queue_entries, only: [:index] do
          member { patch :call; patch :serve; patch :complete; patch :no_show }
        end
      end
    end
  end

Wait estimate calculation:
  estimated_wait_minutes for entry at position N =
    sum of estimated_duration_minutes for all entries ahead (status: waiting/called/in_service)
    Falls back to a default (30 min) if estimated_duration_minutes not set on service category.

Real-time updates (Turbo Streams):
  Queue show page (driver-facing) subscribes to Turbo Stream broadcasts.
  QueueEntry status changes and position updates are broadcast via:
    Turbo::StreamsChannel.broadcast_replace_to("queue_#{queue.id}", ...)
  This is the primary real-time UX — driver sees their position update live without reload.

Seeds: create sample open queue for each seeded workshop.

       test/models/queue_test.rb, test/models/queue_entry_test.rb,
       test/controllers/workshop_management/queues_controller_test.rb

PHASE 14 — MAILERS & NOTIFICATIONS (P0)
----------------------------------------
The original plan created every async flow (workshop approval, service
request state machine, car transfer approval link, queue call) but never
specified how users learn about state changes. Without this:
  - Car transfers are literally unusable — they rely on an emailed token
    link that no mailer produces.
  - Service requests sit unanswered because operators have no inbound
    trigger to check the workshop dashboard.
  - "Driver is called from queue" never reaches the driver.
  - Workshop submitters never learn if they were approved or declined.

Deliverables:
  - ActionMailer configured (SMTP via ENV, default_url_options for
    development + production, async delivery via SolidQueue)
  - ApplicationMailer base + Ukrainian email layout
  - WorkshopMailer: approved, declined (with reason)
  - ServiceRequestMailer: created (to workshop operators),
    status_changed (to driver: accepted / rejected / started / completed)
  - CarTransferMailer: requested (token link to from_user),
    status_changed (approved / rejected / cancelled / expired)
  - QueueMailer: called (driver notified it is their turn)
  - Notification model + NotificationsController — in-app inbox as a
    backup channel for users who do not check email
  - Bell icon in application layout with unread-count badge

Every email delivery runs async via SolidQueue (deliver_later).


PHASE 15 — REAL-TIME TURBO STREAMS (P0)
----------------------------------------
QueueEntry has broadcast stubs but there is no end-to-end wiring
verified in production: cable.yml is not configured for Redis, no
turbo_stream_from tag exists in the driver-facing queue view, and no

update live" is this product's single biggest differentiator — it must
actually work.

Deliverables:
  - cable.yml configured for Redis (async in dev/test, redis in prod)
  - ActionCable reachable from import-map Turbo client
  - turbo_stream_from "queue_#{queue.id}" in queue_entries/show.html.erb
  - QueueEntry#after_update_commit broadcast_replace_to to the queue
    stream (position change, status change)
  - Workshop management queue show subscribes to the same stream
    (operator sees driver join live, no reload)
  - ServiceRequest broadcasts status changes to the driver's request
    show page (status flips live)
  - End-to-end system test: two browsers, operator calls driver,
    driver sees status flip "waiting" → "called" without reload


PHASE 16 — REVIEWS & RATINGS (P1)
----------------------------------
The product vision explicitly lists "reviews, ratings, verified business
profiles" as a quality bar, but the original plan has no review model.
For a marketplace this is load-bearing: without it, drivers have no
trust signal and workshops have no earned reputation.

Data model:
  - Review belongs_to :user, :workshop, :service_request
    unique index on service_request_id — one review per completed service
    rating  integer 1..5, not null
    body    text, nullable
  - Workshop caches avg_rating (decimal 3,2) and review_count (integer),
    updated via after_save on Review

Flow:
  Driver completes a service → email + in-app prompt "Rate this workshop"
  → review form (rating + optional body) → submits → workshop rating
  recalculates. Reviews displayed on workshop show page (list, newest
  first) and aggregate star rating + count on workshop index cards.


PHASE 17 — SEARCH, SORT, PAGINATION (P1)
-----------------------------------------
Current discovery: filter by city/category/open_now and a bounding-box
proximity filter. That does not meet "find a tire shop in under 60
seconds" from the product vision. Gaps:
  - No text search on workshop name or address

    not come first

Deliverables:

  - Pagination on every index: workshops, service_requests, cars, admin
